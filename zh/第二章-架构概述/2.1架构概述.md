# Nanobot 核心架构与模块详解

## 一、架构总览

Nanobot 采用经典的**分层架构设计**，通过消息总线实现各模块间的异步解耦。整个系统可以划分为四个核心层次：

1. **接入层（Channels）**：负责与外部聊天平台交互
2. **总线层（Bus）**：异步消息路由与解耦
3. **核心层（Agent Core）**：AI 推理与工具执行引擎
4. **基建层（Infrastructure）**：LLM 提供者与数据持久化

## 三、模块详解

### 3.1 Agent 模块 —— 智能体大脑 🧠

**位置**：`nanobot/agent/`

Agent 模块是整个系统的核心，负责接收消息、调用 LLM、执行工具、管理记忆。它采用经典的 **ReAct（Reasoning + Acting）循环** 架构。

#### 核心子模块：

| 文件 | 职责 | 代码规模 |
|------|------|----------|
| `loop.py` | **Agent 主循环** —— 接收消息、调用 LLM、执行工具、返回响应 | ~500 行 |
| `context.py` | **上下文构建器** —— 组装 System Prompt、历史消息、记忆、技能 | ~200 行 |
| `memory.py` | **记忆系统** —— 三层记忆模型（Session/History/Memory） | ~300 行 |
| `skills.py` | **技能管理器** —— 加载和管理用户自定义技能 | ~150 行 |
| `subagent.py` | **子代理管理器** —— 支持后台任务和并发执行 | ~200 行 |
| `tools/` | **工具集** —— 文件操作、Shell 执行、网络搜索等 | ~800 行 |

#### Agent Loop 核心流程：

```python
# agent/loop.py 简化版核心逻辑
async def run(self) -> None:
    """Agent 主循环"""
    while self._running:
        # 1. 从消息总线获取入站消息
        msg = await self.bus.consume_inbound()
        
        # 2. 分发消息处理任务
        task = asyncio.create_task(self._dispatch(msg))

async def _process_message(self, msg: InboundMessage) -> None:
    """处理单条消息"""
    # 3. 获取/创建会话
    session = self.sessions.get_or_create(msg.session_key)
    
    # 4. 构建上下文（System Prompt + 历史 + 记忆 + 技能）
    messages = self.context.build_messages(session, msg.content)
    
    # 5. 运行 ReAct 循环
    response, tools_used = await self._run_agent_loop(messages)
    
    # 6. 发送响应到消息总线
    await self.bus.publish_outbound(OutboundMessage(...))
```

### 3.2 Bus 模块 —— 消息总线 🚌

**位置**：`nanobot/bus/`

Bus 模块是系统的**异步通信中枢**，通过双向队列实现 Channels 与 Agent 之间的解耦。

#### 核心组件：

| 文件 | 职责 | 关键特性 |
|------|------|----------|
| `queue.py` | **消息队列** —— 基于 asyncio.Queue 的双向队列 | 仅 ~45 行代码 |
| `events.py` | **事件定义** —— InboundMessage/OutboundMessage 数据类 | 类型安全 |

#### 消息流向：

```
┌─────────────┐     ┌──────────────┐     ┌─────────────┐     ┌─────────────┐
│   用户消息   │────▶│   Channel    │────▶│ Inbound     │────▶│   Agent     │
│  (WhatsApp) │     │  (WhatsApp)  │     │   Queue     │     │   Loop      │
└─────────────┘     └──────────────┘     └─────────────┘     └──────┬──────┘
                                                                     │
                              ┌──────────────────────────────────────┘
                              │
                              ▼
┌─────────────┐     ┌──────────────┐     ┌─────────────┐     ┌─────────────┐
│   用户收到   │◀────│   Channel    │◀────│ Outbound    │◀────│  LLM 响应   │
│    回复     │     │  (WhatsApp)  │     │   Queue     │     │  (Agent)    │
└─────────────┘     └──────────────┘     └─────────────┘     └─────────────┘
```

#### 核心代码：

```python
# bus/events.py
@dataclass
class InboundMessage:
    """从聊天渠道接收的消息"""
    channel: str           # 渠道名称: telegram, whatsapp, feishu...
    sender_id: str         # 发送者标识
    chat_id: str           # 聊天/频道标识
    content: str           # 消息内容
    timestamp: datetime    # 时间戳
    media: list[str]       # 媒体文件 URL 列表
    metadata: dict         # 渠道特定数据

@dataclass
class OutboundMessage:
    """要发送到聊天渠道的消息"""
    channel: str
    chat_id: str
    content: str
    reply_to: str | None   # 回复哪条消息
    media: list[str]
```

### 3.3 Channels 模块 —— 多渠道适配 📱

**位置**：`nanobot/channels/`

Channels 模块负责对接各种聊天平台，将所有平台的消息统一封装为标准格式。

#### 支持的平台：

| 平台 | 文件 | 特点 |
|------|------|------|
| **Telegram** | `telegram.py` | Bot API，长轮询 |
| **WhatsApp** | `whatsapp.py` | WhatsApp Business API |
| **Discord** | `discord.py` | Bot Gateway |
| **飞书 (Feishu)** | `feishu.py` | WebSocket 长连接 |
| **钉钉** | `dingtalk.py` | Webhook + 机器人 |
| **QQ** | `qq.py` | OneBot 协议 |
| **Slack** | `slack.py` | Bot Token |
| **Email** | `email.py` | IMAP/SMTP |
| **企业微信** | `wecom.py` | 企业微信 API |

#### 架构设计：

```python
# channels/base.py - 抽象基类
class BaseChannel(ABC):
    """聊天渠道抽象基类"""
    name: str = "base"
    
    def __init__(self, config: Any, bus: MessageBus):
        self.config = config
        self.bus = bus
    
    @abstractmethod
    async def start(self) -> None:
        """启动渠道监听"""
        pass
    
    @abstractmethod
    async def send(self, message: OutboundMessage) -> None:
        """发送消息到渠道"""
        pass
    
    async def _handle_message(self, msg: InboundMessage) -> None:
        """统一消息处理入口"""
        await self.bus.publish_inbound(msg)
```

#### 设计优势：

- **统一接口**：所有渠道实现相同的 `BaseChannel` 接口
- **自动发现**：通过 `pkgutil` 自动发现新渠道，无需手动注册
- **权限控制**：支持 `allow_from` 白名单，防止未授权访问

### 3.4 Providers 模块 —— LLM 提供者 🤖

**位置**：`nanobot/providers/`

Providers 模块提供统一的 LLM 调用接口，支持多种模型提供商。

#### 支持的提供商：

| 提供商 | 说明 |
|--------|------|
| **OpenRouter** | 统一 API 网关，支持 100+ 模型 |
| **Anthropic (Claude)** | Claude 3 系列 |
| **OpenAI** | GPT-4 / GPT-3.5 |
| **DeepSeek** | 国产开源模型 |
| **Google (Gemini)** | Gemini Pro |
| **Groq** | 高速推理 API |
| **vLLM** | 本地模型部署 |

#### 核心设计：

```python
# providers/base.py - 统一接口
class LLMProvider(ABC):
    """LLM 提供者抽象基类"""
    
    @abstractmethod
    async def chat(
        self,
        messages: list[dict],
        tools: list[dict] | None = None,
        model: str | None = None,
    ) -> LLMResponse:
        """调用 LLM 进行对话"""
        pass
    
    @abstractmethod
    def get_default_model(self) -> str:
        """获取默认模型名称"""
        pass
```

**位置**：`nanobot/agent/tools/`

#### 内置工具：

| 工具 | 功能 | 用途 |
|------|------|------|
| `read_file` | 读取文件内容 | 代码分析、文档阅读 |
| `write_file` | 写入文件 | 代码生成、文档创建 |
| `edit_file` | 编辑文件 | 代码修改 |
| `list_dir` | 列出目录内容 | 文件浏览 |
| `exec` | 执行 Shell 命令 | 系统操作 |
| `web_search` | 网络搜索 | 信息检索 |
| `web_fetch` | 获取网页内容 | 网页分析 |
| `message` | 发送消息 | 主动通知 |
| `spawn` | 创建子代理 | 并发任务 |
| `cron` | 定时任务 | 任务调度 |

#### 工具注册机制：

```python
# tools/registry.py
class ToolRegistry:
    """工具注册表"""
    
    def __init__(self):
        self._tools: dict[str, BaseTool] = {}
    
    def register(self, tool: BaseTool) -> None:
        """注册工具"""
        self._tools[tool.name] = tool
    
    def get_definitions(self) -> list[dict]:
        """获取工具定义（用于 Function Calling）"""
        return [tool.get_definition() for tool in self._tools.values()]
    
    async def execute(self, name: str, arguments: dict) -> str:
        """执行工具"""
        if tool := self._tools.get(name):
            return await tool.execute(**arguments)
        return f"Error: Tool '{name}' not found"
```

### 3.7 Cron 模块 —— 定时任务 ⏰

**位置**：`nanobot/cron/`

Cron 模块支持定时任务调度，让 Agent 可以在指定时间自动执行某些操作。

- 每天早上 8 点发送日报
- 每小时检查一次系统状态
- 定时备份数据

---

### 3.8 Heartbeat 模块 —— 心跳机制 💓

**位置**：`nanobot/heartbeat/`

Heartbeat 模块实现周期性唤醒机制，让 Agent 可以主动检查待处理任务。

#### 工作原理：

1. 定期读取 `HEARTBEAT.md` 文件
2. 检查是否有待执行的任务
3. 如有任务，触发 Agent 执行
4. 如无任务，回复 `HEARTBEAT_OK`

## 五、设计哲学

### 5.1 显式优于隐式

Nanobot 的设计遵循 Python 之禅——"显式优于隐式"。每个功能都写在明处，没有复杂的依赖注入或魔法方法。

### 5.2 简单够用

- **不用向量数据库**：用 `grep` 搜索记忆文件，简单高效
- **不用复杂 ORM**：直接用 JSONL 存储会话，一目了然
- **不用重量级框架**：纯 asyncio + 标准库，轻量可控

### 5.3 模块化扩展

每个模块都有清晰的接口，新增功能只需：

1. 新增渠道？继承 `BaseChannel`
2. 新增工具？继承 `BaseTool`
3. 新增 LLM？继承 `LLMProvider`

## 六、与 OpenClaw 的架构对比

| 维度 | OpenClaw | Nanobot |
|------|----------|---------|
| **架构复杂度** | 高度模块化，但耦合严重 | 扁平分层，职责清晰 |
| **代码组织** | 43 万行，多层抽象 | 4,000 行，一目了然 |
| **消息机制** | 复杂的内部事件系统 | 简单的 asyncio Queue |
| **扩展方式** | 插件市场 + 动态加载 | 静态注册 + 代码扩展 |
| **部署方式** | Docker + 多服务 | 单进程 + 可选容器 |

---

## 七、总结

Nanobot 的架构设计体现了"少即是多"的哲学：

1. **消息总线解耦**：Channels 与 Agent 通过 Queue 通信，互不影响
2. **统一接口设计**：所有渠道、工具、LLM 都遵循标准接口
3. **分层清晰**：接入层、总线层、核心层、基建层各司其职
4. **扩展友好**：新增功能只需实现对应接口，无需改动核心

这种设计让 Nanobot 既保持了极简的代码量，又具备了生产环境所需的稳定性和可扩展性。
